# Fragile

**Find the files you are afraid to touch.**

Fragile analyzes your codebase to find load-bearing, undocumented, high-risk files. It uses structural signals (change frequency, dependency count, test coverage) to identify risky code, then uses AI to explain in plain English why each flagged file is dangerous.

## Quick Start

> **Warning**
> **`fetch-depth: 0` is REQUIRED.** Without full git history, Fragile cannot accurately count file changes over the last 90 days. This is the #1 setup mistake and will cause inaccurate results.

```yaml
name: Fragile Analysis

on:
  schedule:
    - cron: '0 0 * * 1'  # Weekly on Monday
  workflow_dispatch:

jobs:
  fragile:
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0  # <-- REQUIRED! Do not remove.

      - uses: patchwork-eng/fragile@v1
        with:
          openai_key: ${{ secrets.OPENAI_API_KEY }}
```

## Inputs

| Input | Required | Default | Description |
|-------|----------|---------|-------------|
| `openai_key` | Yes | - | Your OpenAI API key |
| `license_key` | No | - | License key (required for private repos) |
| `top_n` | No | `10` | Number of fragile files to surface |
| `min_references` | No | `3` | Minimum reference count to include a file |
| `coverage_path` | No | - | Path to lcov.info or coverage-summary.json |

## How It Works

Fragile calculates a risk score for each file based on:

1. **Change frequency** (40%): How often the file has been modified in the last 90 days
2. **Reference count** (40%): How many other files import/require this file
3. **Coverage gap** (20%): How much of the file is untested (if coverage data provided)

Files are ranked by risk score, and the top N are flagged. For each flagged file, GPT-4o-mini generates a plain-English explanation of why it's dangerous.

## Example Output

Here's what a generated `FRAGILE.md` looks like:

```markdown
# FRAGILE.md
> Generated by [Fragile](https://fragile.dev) — 2024-01-15

These files are load-bearing. Change them carefully.

## 🔴 src/core/auth.js
**Risk score: 92** · 47 dependents · changed 23 times in 90 days · 12% test coverage

This authentication module is the single point of entry for all user sessions. Nearly every route depends on it for token validation. A bug here could lock out all users or, worse, create security vulnerabilities that expose user data.

---

## 🟠 src/utils/database.js
**Risk score: 71** · 31 dependents · changed 15 times in 90 days · 45% test coverage

This database utility handles all connection pooling and query execution. Multiple services depend on its retry logic and connection management. Failures here cascade to complete application unavailability.

---

## 🟡 src/api/router.js
**Risk score: 58** · 22 dependents · changed 8 times in 90 days · 67% test coverage

The central routing file that maps all API endpoints. Changes here can accidentally break endpoint paths or middleware chains, affecting multiple features simultaneously.

---
```

## Advanced Configuration

### With Coverage Data

```yaml
- uses: patchwork-eng/fragile@v1
  with:
    openai_key: ${{ secrets.OPENAI_API_KEY }}
    coverage_path: coverage/lcov.info
```

### Custom Thresholds

```yaml
- uses: patchwork-eng/fragile@v1
  with:
    openai_key: ${{ secrets.OPENAI_API_KEY }}
    top_n: 20
    min_references: 5
```

### Private Repository

```yaml
- uses: patchwork-eng/fragile@v1
  with:
    openai_key: ${{ secrets.OPENAI_API_KEY }}
    license_key: ${{ secrets.FRAGILE_LICENSE_KEY }}
```
